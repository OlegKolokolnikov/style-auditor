# Style Auditor

Инструмент для обнаружения паттернов ИИ-генерации в русскоязычных текстах. Подсвечивает стилистические признаки — конструкции, ритм и лексику, которые непропорционально часто встречаются в текстах, написанных языковыми моделями — и выдаёт **индекс сглаженности** с пооценочными баллами по каждому фрагменту.

Инструмент не утверждает, что текст написан ИИ. Он указывает на места, заслуживающие ручной проверки.

---

## Как это работает

Текст разбивается на чанки (~1100 символов, с разбивкой по абзацам или предложениям). Каждый чанк прогоняется через 23 независимые проверки двух категорий:

**Паттерн-чеки** — поиск конкретных конструкций, характерных для ИИ-текста:
- Контрастные рамки: «Это не X. Это Y.», «Не потому что X, а потому что Y.»
- Шаблонные связки: «нечто большее, чем», «не просто X, а»
- Стилистические клише: псевдоафоризмы, гиперболические усилители, помпезные сравнения
- Структурные признаки: тройные перечисления, два прилагательных перед существительным, абзацы-крючки из одного слова, повторы начал предложений, серии коротких фраз

**Сглаженность-чеки** — агрегированные метрики по чанку:
- **Лексическая предсказуемость** — доля частотных и абстрактных слов, эмоциональных маркеров и сглаженных связок по сравнению с уникальными и длинными словами
- **Монотонность ритма** — дисперсия длин предложений; низкая дисперсия сигнализирует об однообразном, характерном для ИИ ритме
- **Общие слова** — избыток размытых прилагательных (красивый, странный, тёмный и т.п.)

Каждая проверка вносит взвешенный вклад в итоговый балл. Баллы по чанкам ограничены диапазоном 0–100 и усредняются по всему тексту для получения **индекса сглаженности**.

### Уровни риска

| Балл | Метка чанка | Итоговая метка |
|------|-------------|----------------|
| 0–34 | Низкий риск | Низкая сглаженность |
| 35–59 | Умеренный риск | Умеренная сглаженность |
| 60–100 | Высокий риск сглаженности | Высокая сглаженность |

## Локальный запуск

### Требования

- Java 21+
- Maven 3.9+
- Node.js 20+

### Backend

```bash
cd backend
mvn spring-boot:run
```

API доступен по адресу `http://localhost:8080/api/analyze`.

### Frontend

```bash
cd frontend
npm install
npm run dev
```

Интерфейс доступен по адресу `http://localhost:5173`.

### Docker Compose (оба сервиса вместе)

```bash
docker compose up --build
```

- Frontend: http://localhost:5173
- Backend: http://localhost:8080/api/analyze

## Деплой на Render

В репозитории есть `render.yaml` Blueprint, описывающий оба сервиса.

1. Загрузите проект в GitHub-репозиторий.
2. В Render нажмите **New → Blueprint** и укажите репозиторий.
3. Render создаст два сервиса: `style-auditor-backend` и `style-auditor-frontend`.
4. После первого деплоя задайте переменные окружения:

**Сервис backend:**
```
FRONTEND_ORIGIN=https://<адрес-фронтенда>.onrender.com
```

**Сервис frontend (запускает повторный деплой):**
```
VITE_API_URL=https://<адрес-бекенда>.onrender.com/api/analyze
```

> На бесплатном тарифе Render сервисы засыпают при простое. Первый запрос после паузы может занять 10–30 секунд.

### Деплой frontend на Vercel или Netlify

Корневая директория: `frontend`  
Команда сборки: `npm ci && npm run build`  
Директория публикации: `dist`  
Переменная окружения: `VITE_API_URL=https://<адрес-бекенда>/api/analyze`

## API

```
POST /api/analyze
Content-Type: application/json

{ "text": "..." }
```

Возвращает `AnalysisResult` с полями:

- `smoothnessIndex` — итоговый балл от 0 до 100
- `smoothnessLabel` — текстовая метка уровня сглаженности
- `verdict` — одно предложение с рекомендацией для ревьюера
- `summary` — количество чанков, подозрительных чанков, топ проблем
- `chunks[]` — разбивка по чанкам с `flags`, `highlights`, `riskScore`, `lexicalPredictability`, `rhythmMonotony`

Каждый `highlight` содержит смещения `start`/`end` в символах относительно текста чанка и уровень `severity` (`low` / `medium` / `high`). Фронтенд использует эти данные для отрисовки цветных тегов `<mark>`.

## Добавление новой проверки

Создайте класс в `backend/src/main/java/com/styleauditor/checks/`, реализующий интерфейс `TextCheck`:

```java
public class MyCheck implements TextCheck {
    @Override
    public CheckResult check(ChunkContext context) {
        CheckResult result = new CheckResult();
        // context.text()              — текст чанка
        // context.sentences()         — список предложений
        // context.sentencePositions() — список [start, end] смещений для каждого предложения
        // context.sentenceLengths()   — количество слов в каждом предложении
        return result;
    }
}
```

Для проверок на основе регулярных выражений унаследуйтесь от `RegexCheck` и реализуйте методы `pattern()`, `type()`, `severity()`, `comment()` и `weight()`.

Затем добавьте новый класс в список `checks` в `AnalysisService.java`.

Флюентный API `CheckResult`:

```java
result
    .addFlag("Название", "medium", count, "Пояснение, которое видит пользователь.", weight)
    .addHighlight(start, end, "Название", "medium")
    .addSuggestion("Как это исправить.");
```
